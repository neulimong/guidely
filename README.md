## 문제 정의

운영관리·행정 업무를 직접 경험하면서 반복적으로 마주친 문제들에서 출발했습니다.

- **인수인계가 구두로 이루어짐** — 문서가 없으니 담당자가 바뀌면 지식이 함께 사라짐
- **업무 매뉴얼이 여러 파일에 흩어져 있음** — 어디에 어떤 규정이 있는지 찾는 데만 시간을 씀
- **담당자마다 업무 방식과 답변이 다름** — 같은 질문에도 사람마다 다른 답변을 해 혼선 발생
- **신입 직원은 같은 질문을 반복함** — 담당자도 지치고 신입도 눈치가 보이는 구조

> 이 경험에서 "문서를 한 곳에 모으고 AI가 일관된 답변을 주면 어떨까?"라는 질문이 시작됐고, 이 프로젝트로 이어졌습니다.

## 핵심 기능

| 기능 | 설명 |
|------|------|
| 🔍 **AI 문서 검색** | 자연어 질문으로 사내 문서를 검색하고 관련 내용을 요약해서 제공 (RAG) |
| 📋 **SOP 생성** | 업무명을 입력하면 표준 운영 절차서(SOP)를 자동으로 초안 생성 |
| ✅ **체크리스트 생성** | 업무 설명을 입력하면 단계별 체크리스트를 자동 작성 |
| 💬 **사내 문의 답변** | 부서 간 문의나 직원 요청에 대해 회사 규정 기반의 답변 초안을 생성 |

## 시스템 아키텍처

### 문서 업로드 파이프라인

```
문서 업로드 (.txt)
    → 텍스트 추출
    → 청킹 (\n 단위 분리)
    → 임베딩 생성 (gemini-embedding-2, 3072차원)
    → pgvector에 벡터 저장 (Supabase)
```

### 질문 응답 파이프라인 (RAG)

```
질문 입력
    → 질문 임베딩 생성 (gemini-embedding-2)
    → pgvector 유사도 검색 (cosine similarity, threshold 0.5)
    → 상위 5개 청크 컨텍스트로 조합
    → Gemini (gemini-2.5-flash)로 최종 답변 생성
```

## 기술 선택 이유

- **Gemini API** — 한국어 처리 성능이 우수하고, 개발 초기 단계에서 무료 티어로 충분히 검증 가능했음
- **Supabase + pgvector** — Auth, DB, 벡터 검색을 하나의 플랫폼에서 해결할 수 있어 초기 스케일에 최적
- **React + Vite** — 빠른 HMR과 TypeScript 지원으로 개발 속도와 안정성을 동시에 확보
- **gemini-embedding-2** — 3072차원의 고밀도 벡터로 의미 유사도 정확도가 높음

## 트러블슈팅

### 1. pgvector 인덱스 2000차원 제한 문제

**문제**: `gemini-embedding-2`의 임베딩 차원(3072)이 pgvector HNSW/IVFFlat 인덱스의 지원 한계(2000차원)를 초과해 인덱스 생성 불가.  
**해결**: 초기 데이터 규모에서는 인덱스 없이 순차 스캔(sequential scan)으로 운영. 데이터가 수만 건 이상으로 늘어나면 차원 축소(PCA) 또는 다른 임베딩 모델 검토 필요.

### 2. Gemini 무료 플랜 분당 요청 한도 초과

**문제**: 문서 업로드 시 청크별로 임베딩 API를 개별 호출하다가 분당 한도(RPM)에 걸려 업로드 실패.  
**해결**: 단건 호출 방식을 배치 임베딩으로 변경. API 호출 횟수를 14회 → 1회로 줄여 한도 문제 해소 및 업로드 속도 개선.

### 3. 청킹 불량 — chunk_index가 전부 0

**문제**: 업로드한 문서의 모든 청크가 `chunk_index = 0`으로 저장됨. 확인해보니 줄바꿈 문자를 제대로 인식하지 못해 문서 전체가 하나의 청크로 처리되고 있었음.  
**해결**: 청킹 로직을 `\n\n` 단위 분리에서 `\n` 단위 분리로 수정. 이후 청크가 정상적으로 분리되어 저장됨.

## 회고

### 잘한 점
- **문제 정의가 명확했다** — 직접 겪은 운영 업무의 불편함에서 출발해서 "왜 만드는가"에 흔들림이 없었고, 기능 범위를 쉽게 결정할 수 있었다.
- **RAG 파이프라인을 처음부터 끝까지 직접 구현했다** — 문서 파싱, 청킹, 임베딩, 벡터 저장, 유사도 검색, 프롬프트 조합까지 각 단계를 손으로 만들면서 전체 흐름을 체득했다.

### 아쉬운 점
- **청킹 전략이 단순하다** — 현재는 줄바꿈 기준으로만 자르는데, 문서 구조를 고려한 시맨틱 청킹을 적용했다면 검색 정확도가 더 높았을 것 같다.
- **평가 지표가 없다** — 답변이 "좋다/나쁘다"를 사람이 눈으로만 판단하고 있어서, Precision/Recall 같은 RAG 평가 메트릭을 붙였으면 개선 방향이 더 명확했을 것 같다.

### 배운 점
- **청킹은 RAG의 가장 중요한 변수다** — 임베딩 모델이나 LLM보다 청크가 어떻게 나뉘느냐가 검색 품질에 훨씬 직접적인 영향을 준다는 걸 직접 경험했다.
- **프리 티어 제약이 곧 설계 제약이다** — API 호출 한도, 벡터 인덱스 차원 제한 등 인프라 제약을 먼저 파악하고 설계에 반영하는 습관의 중요성을 깨달았다.
