# Guidely — 포트폴리오 소개

---

## 1. 왜 만들었나 (Why)

운영관리·행정 업무를 직접 담당했던 경험이 있습니다.

그 시절 가장 자주 들었던 질문은 항상 비슷했습니다.
"이 서류 어디에 내요?", "법인카드 정산은 어떻게 하나요?", "회의실은 어떻게 예약해요?"

처음엔 친절하게 답했지만, 같은 질문이 반복될수록 비효율이 눈에 띄었습니다.
담당자가 바뀌면 노하우가 사라졌고, 매뉴얼은 여러 파일에 흩어져 있었고,
사람마다 다르게 안내하는 일도 생겼습니다.

그래서 직접 체크리스트를 만들고, 상황별 응대 스크립트를 정리했습니다.
반복되는 질문을 한 장짜리 FAQ로 만들어 공유폴더에 올려두기도 했습니다.
그게 나름의 해결책이었지만, 한계가 있었습니다. 사람이 직접 관리해야 했으니까요.

"이걸 AI로 자동화하면 어떨까?"

그 질문이 이 프로젝트의 시작이었습니다.

---

## 2. 문제 정의 (Problem)

### 페인포인트 3가지

**① 지식이 사람에게 묶여 있다**
업무 노하우가 문서가 아닌 담당자 머릿속에만 존재합니다.
인수인계는 대부분 구두로 이뤄지고, 담당자가 바뀌는 순간 맥락이 사라집니다.

**② 문서가 있어도 찾을 수 없다**
매뉴얼은 있습니다. 하지만 공유드라이브 어딘가에, 이메일 첨부파일로, 메신저 대화 속에 흩어져 있습니다.
무엇이 최신 버전인지도 불분명합니다.

**③ 같은 질문, 다른 답변**
담당자마다 숙지하고 있는 규정의 범위가 다릅니다.
결과적으로 동일한 질문에 다른 답변이 돌아오고, 이는 신뢰 문제로 이어집니다.

### 기존 해결 방법의 한계

가장 흔한 대안은 "FAQ 문서 만들기"입니다. 하지만 이 방식은 유지보수에 사람이 계속 투입돼야 하고,
문서가 많아질수록 오히려 어디서 찾아야 할지 모르는 상황이 반복됩니다.
ChatGPT에 물어보는 방법도 있지만, 회사 규정을 모르는 AI가 지어낸 답변은 오히려 위험합니다.

---

## 3. 내가 선택한 해결 방법 (Solution)

### RAG를 선택한 이유

단순히 AI를 붙이는 것이 목표가 아니었습니다.
회사마다 다른 규정, 부서마다 다른 프로세스에 맞는 답변을 줄 수 있어야 했습니다.

일반 ChatGPT와의 차이는 명확합니다.

| 구분 | 일반 AI (ChatGPT 등) | 이 프로젝트 (RAG) |
|------|----------------------|-------------------|
| 학습 데이터 | 범용 인터넷 데이터 | 우리 회사 내부 문서 |
| 답변 근거 | 모델이 학습한 지식 | 실제 업로드된 사내 규정 |
| 할루시네이션 위험 | 높음 | 낮음 (출처 표시 포함) |
| 커스터마이징 | 불가 | 문서 교체만으로 즉시 반영 |

**RAG(Retrieval-Augmented Generation)** 는 질문이 들어오면 먼저 관련 문서를 검색한 뒤,
그 내용을 근거로 AI가 답변을 생성하는 방식입니다.
"지어내지 않고, 문서에 있는 것만 답한다"는 원칙을 구현할 수 있었습니다.

### 기술 스택 선택 이유 (비즈니스 관점)

**Supabase + pgvector** — 별도의 벡터 전용 DB(Pinecone 등)를 추가하지 않고,
기존 PostgreSQL 안에서 벡터 검색까지 해결했습니다.
초기 단계에서 관리 포인트를 줄이는 것이 더 중요하다고 판단했습니다.

**Gemini API** — 한국어 처리 성능과 무료 티어 제공이 결정적이었습니다.
프로토타입 단계에서는 비용 없이 검증할 수 있어야 빠르게 실험할 수 있습니다.

**React + Vite** — 빠른 개발 환경보다 더 중요한 이유가 있었습니다.
실사용자(운영 담당자)는 기술을 모릅니다. 직관적인 UI가 기술보다 먼저였습니다.

## 4. 구현 과정에서 겪은 진짜 어려움 (Challenge)

### ① pgvector 차원 제한 문제

**문제**
`gemini-embedding-2` 모델은 3072차원 벡터를 생성합니다.
그런데 Supabase의 pgvector는 HNSW/IVFFlat 인덱스를 2000차원까지만 지원합니다.
인덱스 없이는 검색 속도가 급격히 느려질 수 있다는 걸 알고 있었습니다.

**시도**
임베딩 차원을 줄이는 PCA 적용을 검토했습니다.
하지만 차원 축소 과정에서 의미 손실이 발생하고, 전처리 파이프라인이 복잡해졌습니다.
다른 임베딩 모델(`text-embedding-004` 등)도 확인했지만, 접근이 제한된 상태였습니다.

**해결**
현재 데이터 규모(수백 개 청크)에서는 인덱스 없이 순차 스캔으로도 충분히 빠르다는 것을 실측으로 확인했습니다.
"지금 당장 필요하지 않은 최적화에 시간을 쓰지 않는다"는 판단이었습니다.
데이터가 수만 건 이상으로 늘어날 경우 차원 축소 또는 모델 교체를 검토하기로 했습니다.

---

### ② API 한도와 배치 최적화

**문제**
초기 구현에서는 문서를 청킹한 뒤 각 청크마다 임베딩 API를 개별 호출했습니다.
14개 청크가 있으면 14번의 API 요청이 발생했고, Gemini 무료 플랜의 분당 요청 한도(RPM)에 반복적으로 걸렸습니다.
업로드 중간에 429 에러가 발생하고 처리가 중단됐습니다.

**시도**
요청 사이에 딜레이를 넣는 방식(2초 대기)을 먼저 시도했습니다.
문제는 해결됐지만, 14개 청크 처리에 30초 이상이 걸렸고 UX가 나빠졌습니다.

**해결**
Gemini 임베딩 API의 배치 처리 기능을 확인했습니다.
여러 텍스트를 하나의 요청으로 묶어서 보낼 수 있었고, API 호출 횟수를 14회 → 1회로 줄였습니다.
처리 속도가 크게 개선됐고 한도 문제도 함께 해소됐습니다.

---

### ③ 청킹 전략 삽질

**문제**
업로드한 문서의 모든 청크가 `chunk_index = 0`으로 저장됐습니다.
벡터 검색은 되는데 항상 동일한 결과만 나왔고, 다양한 내용이 검색되지 않았습니다.

**시도**
처음에는 임베딩 문제라고 판단해 API 응답을 뜯어봤습니다.
그 다음엔 Supabase 저장 로직을 의심해서 INSERT 쿼리를 콘솔로 출력해봤습니다.
실제 원인을 찾는 데 시간이 걸렸습니다.

**해결**
청킹 함수를 들여다보니 `\n\n` 기준으로 분리하고 있었는데,
테스트에 사용한 `.txt` 문서들이 모두 `\n` 단위로만 줄바꿈이 돼 있었습니다.
결과적으로 문서 전체가 하나의 청크로 처리되고 있었던 것입니다.
분리 기준을 `\n`으로 바꾸고, 10자 미만의 빈 줄은 필터링하는 로직을 추가했습니다.
이후 14개 청크가 정상적으로 생성되고 저장됐습니다.

> 배운 것: 디버깅은 가정이 아니라 데이터에서 시작해야 합니다.
> "임베딩 문제겠지"라는 가정이 원인 파악을 늦췄습니다.

## 5. 결과 (Result)

### 구현된 기능

| 기능 | 설명 |
|------|------|
| 🔍 **AI 문서 검색** | 자연어 질문으로 업로드된 사내 문서를 검색하고 관련 내용을 요약 |
| 📋 **SOP 자동 생성** | 업무명 입력 시 6단계 표준 운영 절차서(SOP) 자동 작성 |
| ✅ **체크리스트 생성** | 상황 설명 입력 시 단계별 체크리스트 생성 + 진행률 표시 |
| 💬 **사내 문의 답변** | 직원 문의에 대해 회사 규정 기반의 답변 초안 자동 생성 |

### 데모 시나리오: 신입 직원의 첫 출근

```
상황: 오늘 첫 출근한 신입 직원 김새벽. 뭘 해야 할지 모른다.
```

**Step 1 — 문서 업로드** (관리자가 미리 수행)
> 문서 업로드 탭 → `신입온보딩_매뉴얼.txt`, `경비처리_규정.txt` 등 사내 문서 업로드
> 자동으로 청킹 → 임베딩 → 벡터 저장 완료

**Step 2 — 첫날 할 일 검색**
> 검색창에 입력: "입사 첫날에 뭘 해야 하나요?"
> AI 답변: "인사팀 방문, 사원증 수령, IT 계정 발급(이메일/ERP/그룹웨어) 순서로 진행하세요."
> 참고 문서: `신입온보딩_매뉴얼.txt` 출처 표시

**Step 3 — 온보딩 체크리스트 생성**
> 체크리스트 탭 → "신입 직원 입사 첫 주 해야 할 일" 입력
> 항목별 체크리스트 자동 생성 → 클릭하면 체크 + 진행률 업데이트

```
결과: 담당자 없이도, 문서만 있으면 신입 직원이 스스로 온보딩 과정을 진행할 수 있다.
```

## 6. 배운 것 (Learning)

### 기술적 배움

**RAG 파이프라인은 LLM보다 청킹이 더 중요하다**
처음에는 어떤 AI 모델을 쓰느냐가 품질을 결정한다고 생각했습니다.
직접 만들어보니 그렇지 않았습니다. 문서가 어떻게 잘리느냐, 즉 청킹 전략이 검색 품질에 훨씬 직접적인 영향을 줬습니다.
좋은 모델도 나쁜 청크 위에서는 나쁜 답변을 냅니다.

**인프라 제약을 먼저 파악해야 한다**
pgvector 차원 제한, API 분당 호출 한도.
이런 제약을 미리 알았다면 처음부터 다른 설계를 했을 것입니다.
앞으로는 구현 전에 사용할 인프라의 제약 조건을 먼저 정리하는 습관을 들이기로 했습니다.

**디버깅은 가정이 아니라 데이터에서 시작한다**
청킹 버그를 찾는 데 예상보다 오래 걸렸습니다.
"아마 이 부분이 문제겠지"라는 가정으로 접근했다가 시간을 낭비했습니다.
콘솔에 데이터를 찍어서 실제 값을 확인하는 것이 가장 빠른 디버깅이었습니다.

### PM·PO 관점 배움

**완벽하지 않아도 배포한다**
인덱스 없이 운영하고, 스크린샷 없이 README를 올리고, 세부 기능 없이 먼저 핵심 흐름을 완성했습니다.
완성을 기다리다 보면 배포를 못 합니다. 동작하는 것을 먼저 만들고, 그 다음에 다듬는 것이 맞습니다.

**기능보다 사용자 경험이 먼저다**
더 많은 기능을 추가하고 싶은 욕심이 계속 생겼습니다.
하지만 기능이 많아진다고 좋은 제품이 되지 않습니다.
"신입 직원이 처음 켰을 때 스스로 사용할 수 있는가?"를 기준으로 판단했고,
그 기준이 기능 범위를 결정하는 데 가장 유효했습니다.
